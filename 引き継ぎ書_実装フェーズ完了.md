## このドキュメントについて

「設計・API準備フェーズ」の引き継ぎ書(`引き継ぎ書` 初版、チャット履歴参照)を踏まえ、
**データ取得スクリプトの実装〜GitHub Actions化までの土台を完了**した。

実装物一式は `fitness-sync/` ディレクトリ配下にある。

## 1. 実装した内容(完了)

### ディレクトリ構成

```
fitness-sync/
├── requirements.txt          # 依存パッケージ
├── .env.example              # 環境変数テンプレート(実体の .env は要作成・gitignore対象)
├── .gitignore
├── notion_writer.py          # Notion書き込み共通ヘルパー(upsertによる冪等性担保)
├── oura_auth.py              # Oura OAuth2 初回認証(手元PCで一度だけ実行)
├── oura_sync.py              # Oura → デイリーコンディションDB 同期
├── garmin_sync.py            # Garmin → ワークアウトDB 同期
└── .github/workflows/
    └── daily_sync.yml        # 毎日自動実行するGitHub Actions workflow
```

### 各スクリプトの役割

**[notion_writer.py](fitness-sync/notion_writer.py)**
- `upsert_page()`: 「日付」(必要なら「日付」+「名前」)が一致する既存ページを検索し、
  あれば更新・なければ新規作成する。これがcron再実行時の**二重登録防止(冪等性)**の要。
- ワークアウトDBのように1日に複数件(ランニング+筋トレ等)入りうるDBは
  `match_by_title=True` を指定することで、日付だけでなくアクティビティ名でも一致判定する。

**[oura_auth.py](fitness-sync/oura_auth.py)**
- OAuth2 authorization code フローの初回認証用。`localhost:8080/callback` でローカルサーバを立て、
  ブラウザでOuraの認可画面を開き、認可コード→アクセストークン+リフレッシュトークンを取得する。
- **これは手元PCで一度だけ実行する**。取得したリフレッシュトークンを `.env` /
  GitHub Secrets の `OURA_REFRESH_TOKEN` に転記する。

**[oura_sync.py](fitness-sync/oura_sync.py)**
- リフレッシュトークンでアクセストークンを取得 → 前日分の `daily_sleep` /
  `daily_readiness` / `sleep` を取得 → 睡眠スコア・レディネス・HRV・安静時心拍・睡眠時間に整形 →
  デイリーコンディションDBへupsert。
- **リフレッシュトークンのローテーション対応**: Ouraがリフレッシュ時に新しいrefresh_tokenを
  返した場合、`new_oura_refresh_token.txt` に書き出す(workflow側で検知してSecretsを自動更新する設計。後述)。

**[garmin_sync.py](fitness-sync/garmin_sync.py)**
- メール+パスワードでGarmin Connectにログイン(2FA未設定のためそのまま通る)→
  前日分のアクティビティを取得 → `activityType.typeKey` でランニング/筋トレ/その他を判別 →
  ランニングは距離・ペース(分/km)、筋トレは種目名・セット数を
  (`get_activity_exercise_sets` から)ベストエフォートで取得 → ワークアウトDBへupsert。

### GitHub Actions workflow([daily_sync.yml](fitness-sync/.github/workflows/daily_sync.yml))

- `cron: "0 22 * * *"` → UTC 22:00 = **日本時間 翌朝7:00** に毎日自動実行(+手動実行 `workflow_dispatch` も可能)
- Oura同期 → Garmin同期 → リフレッシュトークン更新チェック、の順で実行
- リフレッシュトークンがローテーションされていた場合、`gh secret set` で
  リポジトリの `OURA_REFRESH_TOKEN` Secret を自動的に上書きする
  (このステップだけ `GH_PAT` という別のSecret=Personal Access Tokenで認証する。理由は後述)

## 3. 補足: GH_PAT について(リフレッシュトークン自動更新の仕組み)

OuraのOAuth2はリフレッシュ時に新しいリフレッシュトークンを発行することがある(ローテーション)。
GitHub Actionsの標準トークン(`GITHUB_TOKEN`)には**リポジトリのSecretsを書き換える権限がない**ため、
別途 **Personal Access Token(PAT)** を発行し、`GH_PAT` という名前のSecretとして登録する必要がある。

- GitHubの `Settings → Developer settings → Personal access tokens` で発行
- スコープは「Secrets の読み書き」ができる範囲に絞る(Fine-grained PATであれば対象リポジトリの
  `Secrets: Read and write` 権限のみ付与するのが望ましい)
- このPATを `GH_PAT` としてリポジトリSecretsに登録すれば、workflow内の
  `gh secret set OURA_REFRESH_TOKEN ...` が自動実行される

この設計により、**リフレッシュトークンが更新されてもユーザーが手動で書き換える必要がなくなり、
ハンズオフ運用が成立する**。これが引き継ぎ書(初版)で「OAuthの設計上の肝」と書かれていた部分への回答。

## 4. 既知の留意点・要確認事項(継続)

- **Ouraのメンバーシップ**: Gen3 Ouraリングはサブスクリプションが有効でないとAPI取得不可。
  ステップ3のローカル動作確認時に、`daily_sleep` 等が正常に返るかで間接的に確認できる。
- **タイムゾーン**: 各スクリプトは `date.today() - timedelta(days=1)`(=前日)を対象にしている。
  GitHub Actionsの実行環境はUTCなので、cronを `UTC 22:00`(JST翌7:00)に設定することで
  「日本時間の前日分」が正しく揃うタイミングを狙っている。日付境界のズレが疑われる場合は
  実行時刻とOura/Garmin側のタイムゾーン処理を要確認。
- **筋トレの種目・セット数**: `get_activity_exercise_sets` のレスポンス構造は実機データで
  未検証(ライブラリのソースから推測して実装)。実際のデータで「種目」「セット数」が
  期待通り埋まらない場合は、`garmin_sync.py` の `fetch_exercise_summary()` を
  実データのレスポンス構造に合わせて調整する必要がある。
- **食事記録**: 引き継ぎ書(初版)の通り、引き続き手動(写真→AI解析→Notion入力)の運用。
  自動化スコープ外。

## 5. 日次レポートのLINE送信(追加実装・完了)

毎朝7時(JST)に、前日の運動・回復・食事を総合評価したレポートをJPEGで生成し、LINEに自動送信する機能を追加した。健康意識を高めることが目的。

### 追加・変更したファイル
- **[daily_report.py](fitness-sync/daily_report.py)** — 中心スクリプト。`generate`(データ取得→Claude評価→JPEG描画)と `send`(LINE送信)のサブコマンドを持つ。引数なしで実行するとローカル確認用に両方通す。
- **[line_notify.py](fitness-sync/line_notify.py)** — LINE Messaging API でテキスト+画像をプッシュ送信。
- **[notion_writer.py](fitness-sync/notion_writer.py)** — レポート集計用のクエリ/プロパティ取得ヘルパー(`query_by_date` 等)を追加。
- **reports/** — 生成したJPEGの保存先。**gitignoreせずコミットする**(raw.githubusercontent.com で公開URLにするため)。

### 仕組み
1. **データ取得**: 前日分を3データソース(コンディション/ワークアウト/食事)から取得。
2. **評価**: Claude API(`claude-opus-4-8`、adaptive thinking)で総合スコア・一言サマリー・評価コメント・今日のアクションをJSONで生成。前向きで具体的なコーチング。
3. **描画**: matplotlibで視覚的なJPEGレポート(総合スコアの円、コンディション指標、ワークアウト、PFCバー、コーチコメント)。日本語フォントは環境にあるもの(Win: Yu Gothic / Linux: Noto Sans CJK)を自動選択。
4. **公開**: JPEGをリポジトリにコミット&push → raw URLで公開。
5. **LINE送信**: 公開URLが到達可能になってからテキスト+画像をプッシュ。

### LINEの画像送信の制約(重要)
LINEの画像メッセージは**公開HTTPS URLが必須**(バイナリ直接添付不可)。そのため workflow は **生成 → コミット&push → LINE送信** の3ステップ順で、画像がraw URLで到達可能になってから送信する設計。

### 追加で必要なSecrets(GitHub Actions)
| Secret名 | 値 |
|---|---|
| `ANTHROPIC_API_KEY` | Claude APIキー(課金が少額発生) |
| `LINE_CHANNEL_ACCESS_TOKEN` | LINE Messaging API のチャネルアクセストークン |
| `LINE_USER_ID` | 送信先(自分)のuserId |
| `REPORT_PUBLIC_BASE_URL` | `https://raw.githubusercontent.com/<user>/<repo>/main/fitness-sync/reports` |
| `NOTION_DAILY_CONDITION_DS_ID` / `NOTION_WORKOUT_DS_ID` / `NOTION_MEAL_DS_ID` | 各データソースID(`.env.example`に記載) |

> workflow([daily_sync.yml](fitness-sync/.github/workflows/daily_sync.yml))は cron `0 22 * * *`(UTC22:00 = JST翌7:00)に Oura同期 → Garmin同期 → レポート生成 → JPEGコミット → LINE送信 の順で実行する。日本語フォント(`fonts-noto-cjk`)のapt導入ステップも追加済み。

### 評価のフォールバック(実装済み)
- **`ANTHROPIC_API_KEY` 未設定時はルールベース評価に自動フォールバック**する(`rule_based_evaluation`)。睡眠スコア・レディネス・運動有無・タンパク質量のしきい値でスコアとコメントを生成。**ゼロコスト運用のままにしたい場合はこのSecretを登録しなければよい**。
- Claude API呼び出しが失敗した場合(障害・レート制限等)も同じくルールベースにフォールバックし、レポート送信自体は止まらない。

### 留意点
- `REPORT_PUBLIC_BASE_URL` のブランチ名(`main`等)とパスはリポジトリ構成に合わせること。リポジトリがprivateだとraw URLにLINEがアクセスできない → **publicリポジトリにするか、別の公開ホスティングが必要**。
- Claude APIを使う場合は少額課金が発生(評価1回あたり数円程度)。コストを抑えたい場合は `claude-haiku-4-5` への変更、またはAPIキー未登録のままルールベース運用。
- ローカル確認は `python daily_report.py`(引数なし)で生成〜送信まで通せる。LINE未設定時はテキストを標準出力に表示するだけでスキップ。
- **2026-06-09時点でローカルのエンドツーエンド検証済み**(Notion実データ取得→ルールベース評価→JPEG生成まで成功。`reports/2026-06-09.jpg` 参照)。

### 運用上の注意(トラブル時に読む)
- **Garminはトークンログイン**: データセンターIPからのパスワードログインはGarminにブロックされる(429/MFA)ため、`garmin_auth.py` を手元PCで実行して取得したOAuthトークン(`GARMIN_TOKENS` Secret、約1年有効)を使用。**期限切れ時は `garmin_auth.py` を再実行**(メール認証コード入力あり)→ 新トークンをSecretsに再登録。
- **OuraのリフレッシュトークンはActions側で自動更新される**: そのためローカル `.env` の `OURA_REFRESH_TOKEN` はすぐ古くなる。ローカルで `oura_sync.py` を動かしたい場合は `oura_auth.py` で再認証するか、Secretsの最新値を使うこと。
- LINEのトークン類・PATが漏れた場合は、LINE Developers / GitHub設定で即時再発行し、Secretsを更新する。
